## 내가 필요로 하는 서비스를 만들어라

지난해 테오콘에서 들었던 말이 기억에 남았습니다.

> **"내가 필요로 하는 서비스를 만드세요."**

"내가 꾸준히 사용하는 게 뭔가? 아쉬움을 느끼고 있는 것은 뭔가?"라고 되짚어보니 **기술 블로그**였습니다.

개발을 시작하면서 꾸준히 사용하고 있고, 기존에 사용하던 블로그 서비스에서 여러 아쉬움을 느끼고 있었습니다.

> 프로젝트에서 구현한 기능을 블로그에서 동작할 수 있게 한다면 좋겠다.
>
> 아.. 이런 스타일을 적용하고 싶은데, 왜 됐다가 안됐다가 하지?
>
> 숏폼이 인기를 끌고 AI를 사용해 검색하고 요약하는 시대에, 이제까지의 블로그 형식에 사용자들이 흥미를 느낄까?

마침 우테코 동문분이 AI로 웹툰을 만드는 것을 하고 계셨고, AI 햄스터 유튜브 같은 콘텐츠도 즐겨보고 있었습니다. 그때 생각이 들었습니다. "개발 글도 텍스트만 아니라 웹툰이나 동영상처럼 재미있게 표현할 수 있지 않을까?"

그래서 결정했습니다.

**"직접 만들어보자."** 그리고 **"AI 도구가 정말 좋은지 확인해보자."**

## Stitch - "찰떡같이 알아서 만들어주는 디자이너"

처음엔 피그마로 직접 만들었는데, 너무 오래 걸렸고 디자인이 1차원적이었습니다.
이왕 만드는 것 잘 만들고 싶었습니다.

그러던 중, SNS에서 Stitch를 발견했습니다.

Stitch에게 1차 피그마 디자인, 참고 자료(디자인 템플릿 사이트, 원하는 느낌의 디자인 이미지 등)와 함께 프로젝트의 구조 등을 설명했습니다.

**Stitch가 만들어 준 디자인은 마음에 들었습니다.** 반응형까지 자동으로 구현되었고, 피그마에 바로 붙일 수 있었습니다.

원한다면 동적 UI도 코드로 실행해볼 수 있었고, 프로젝트에 사용되는 테마와 아이콘이 정리된 페이지도 생성해주었습니다. 이런 것까지 해줄 수 있구나 하고 놀랐습니다.

다음 프로젝트에서도 반드시 사용할 것 같습니다.

이번 경험의 교훈을 삼아, **정확한 디자인 용어를 사용해 명령하고, 첫 페이지의 테마와 아이콘을 정리한 후, 이를 기반으로 나머지 페이지를 구현**할 것 같습니다.

Stitch도 개발 AI를 사용할 때와 마찬가지로, 전문 용어를 사용할수록 기본 셋팅을 명확하게 할수록 결과물의 퀄리티가 더 좋았고 이를 바탕으로 개발하는 편의성이 더 좋았습니다.

- ❌ "미니멀하고 세련되게"

- ✅ "산세리프, 중성 톤 베이지 + 코발트 블루, 여백은 16px 이상"

## CodeRabbit - "꼼꼼한 리뷰어"

코드 리뷰 도구로 CodeRabbit을 도입했습니다.

혼자 개발하면서 놓친 부분, 오류를 검토 받고 리뷰를 통해서 성장하고 싶어 도입하게 되었습니다.

그래서 웹 접근성, 보안, 성능 등 다방면으로 검사하고, 꼼꼼하게 리뷰를 하도록 설정했습니다.

**예상보다 훨씬 꼼꼼한 리뷰어**였습니다.

엣지 케이스, 웹 접근성 이슈, 성능 최적화까지 모두 짚어주었습니다. 특히 마크다운 아티클 변환 과정의 잠재적 문제를 지적해, CI에 테스트를 추가했습니다.

## Claude Code - "AI를 사용하는 개발자, 책임자로서의 무게를 견뎌라"

예전에 Claude를 유료 구독한 경험이 있었지만, Claude Code를 사용하는 것은 처음입니다.

Claude Code는 듣던대로 정말 놀라운 생산성을 안겨주었습니다.

**왜일까요?** 전체 사이클을 보면:

```md
1. 설계 문서 작성 ← 예상보다 오래
2. 프롬프트 작성
3. Claude가 코드 작성 ← 빠름
4. 검토 및 피드백 ← 예상보다 오래
5. 수정 및 배포
```

### 설계 단계

Claude에게 "아티클 리스트 페이지 만들어줄래?"라고 하면 뭔가 만듭니다. 하지만 내가 원하는 대로 만들지는 않습니다. 프로젝트 구조, 동작 플로우, 라이브러리 선택 이유, 엣지 케이스 처리 등을 모두 명확히 정의해야 했습니다. 머릿속에 있는 것들을 말로 구체화하는 작업이 꼭 필요했습니다. 코드를 치던 시간들이 이런 작업에 옮겨간 느낌입니다.

### 검토 단계

> **"Claude가 코드를 작성하지만, 책임은 내가 진다."**

AI와 페어 프로그래밍을 한다는 느낌으로 개발을 하고 합니다.
페어 프로그래밍에서는, 각자가 네이게이션과 드라이버의 역할을 나누지만,
AI와의 페어 프로그래밍에서 방향을 설정하는 네비게이터도, 핸들을 잡아 책임지는 드라이버도 모두 개발자 자신이라고 생각합니다. 마치, 자율 자동차가 알아서 주행하지만 운전석에서 핸들을 잡고 있는 것은 운전자인 것 처럼.

그래서 Claude가 짠 코드를 받으면, 단순히 "작동하는가?"만 확인하지 않고, 질문을 통해 검증하고 분석하는 것에 더 집중하는 편입니다.

**🤔 개발 시, Claude의 답변을 보면서 던지는 질문들**

```md
이렇게 구현한 근거는 뭐지? 그 근거가 정말 올바른 지식을 바탕으로 되었는가?
현재 구조에 오버 스택은 아닌가?
확장 가능한가?
더 간단한 방법은?
라이브러리 선택이 맞나?
오류 처리는 충분한가?
```

### AI는 완벽하지 않다.

Tailwind를 사용하는 프로젝트에서 Claude에게 블로그 카테고리별 뱃지의 스타일을 상수로 관리하고 싶다고 명령했습니다. 그 답으로 Claude는 아래의 코드를 제시했습니다.

```jsx
export const CATEGORY_LABELS_COLOR = {
  troubleshooting: { bg: "bg-red-100", text: "text-red-500" },
  //....
};

<div className={CATEGORY_LABELS_COLOR[category].bg}>
```

잘 작동할까요?
일부 색상은 작동하지만, 나머지는 적용되지 않았습니다.

**Tailwind는 빌드 타임에 동작**합니다. 빌드 시, 프로젝트 파일을 스캔하여 사용된 클래스명을 기반으로 CSS를 생성합니다. 그래서 정적으로 작성된 클래스명만 감지할 수 있으며, 변수나 템플릿 문자열을 통해 동적으로 생성되는 클래스명은 감지하지 못할 수 있습니다. 이런 경우 tailwind.config.js의 safelist에 명시적으로 포함해야 합니다.

이런 오류에 대해 지적하니 Claude는 바로 잘못된 코드 구현에 대해 수긍했고 수정된 코드를 제시했습니다.

아마도 Claude가 기존 코드 패턴을 학습해 일관성 있는 네이밍을 선택했지만, Tailwind의 빌드 타임 작동 방식과의 불일치를 놓쳤던 것 같습니다.

AI는 정말 똑똑하고, 똑똑한 만큼 그럴 듯하게 답변합니다. 위와 같은 오류는 로컬 실행 시 바로 오류 상황을 캐치할 수 있지만, 검토 없이 혹은 개발자가 검증할 지식을 가지지 않고 무조건적으로 수용한다면 AI가 짠 오류가 스노우볼처럼 커지는 상황이 발생할 수 있습니다.

## 결론

이번 프로젝트를 통해 가장 강렬하게 느낀 것은 **책임감의 중요성**입니다.

요즘 AI 시대에 개발자의 미래에 대한 여러 이야기를 들었습니다. 그중 가장 공감이 가는 것은 책임자로서의 역할에 관한 것이었는데, 이번 블로그 프로젝트를 구현하면서 직접 경험하고 느낄 수 있었습니다.

우테코에서 빨리 변하는 기술 스택에 대해 코치님에게 조언을 구한 적이 있습니다. 코치님은 **"결국 기본기가 중요하다"**고 말씀해주셨습니다. 새로운 기술도 이전 기술의 토대 위에서 발전하고, 탄탄한 기본기는 이를 빠르게 흡수하는 역량이 된다고요.

눈을 감았다 뜨면 새로운 AI가 나오는 시대라도, **기본기와 개발 지식**이 길을 잃지 않게 할 것 같습니다. 두쫀쿠가 디저트 시장을 흔들어도, 붕어빵은 언제나 스테디 셀러인 것 처럼 말입니다.
